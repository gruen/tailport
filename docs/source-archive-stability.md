# Source archive stability

Why the AUR source package and the Homebrew formula pin a tarball GitHub
generates rather than one we publish — and why that is a known, accepted risk
rather than an oversight. Decided under kata 3qyp.

## What we pin

Two packages pin GitHub's **auto-generated tag archive**:

| File | Field |
| ---- | ----- |
| [`packaging/aur/tailport/PKGBUILD`](../packaging/aur/tailport/PKGBUILD) | `source=` + `sha256sums` |
| [`packaging/brew/tailport.rb`](../packaging/brew/tailport.rb) | `url` + `sha256` |

Both resolve to `https://github.com/gruen/tailport/archive/refs/tags/vX.Y.Z.tar.gz`.

Nobody uploads that file. GitHub produces those bytes on demand by running
`git archive` at request time, and we pin a digest of the result. That is the
whole issue: **the artifact our digests attest to is generated by someone else,
on their schedule, with their tooling.**

`tailport-bin` is **not** affected — it pins the real release assets that
`build.yml` uploads, which are immutable once published.

## What actually happened, once

On **2023-01-30** GitHub deployed Git 2.38 to the service backing source
downloads. That switched `git archive`'s default compressor from an external
`gzip` to an internal implementation, which changed the compressed bytes — and
therefore the checksum — of every `/archive/` tarball, **including tags
published years earlier**. Build systems that pinned digests broke
simultaneously and en masse; Homebrew, Spack, EasyBuild, CMake-based projects
and the AUR all felt it at once. GitHub reverted the change.

On **2023-02-21** GitHub responded with a commitment:

- source downloads held byte-for-byte stable **"for no less than a year"**;
- **six months' notice** in documentation, blog, and changelog before changing
  either archive format — with an explicit carve-out to shorten or omit that
  notice for critical security vulnerabilities;
- no present intent to change either format.

Read the shape of what is left today. The one-year stability window lapsed in
**February 2024** and was not replaced with a standing guarantee. What protects
us now is a *notice policy* plus a *stated intent* — not a contract. The bytes
are still generated on demand by a third party.

## Blast radius if it recurs

Low probability, high blast radius — which is exactly the profile that earns a
written note rather than a fix:

- **Every published version breaks at the same moment**, not just the newest.
  The change is retroactive across all tags.
- **Old versions break for new installs.** `makepkg` re-fetches the tarball on
  every build; there is no upstream cache to fall back on. A user installing
  v0.1.4 a year from now fetches the archive fresh and validates our stale
  digest.
- **It fails silently until a user reports it.** Nothing in CI re-validates the
  digest of an already-published version, so we would learn from a bug report.
- `tailport-bin` keeps working throughout, which is worth remembering during
  triage: "the -bin package installs fine" is consistent with this failure, not
  evidence against it.

## The decision: accept it

Do nothing until it happens; fix reactively if it ever does. The reasoning:

- The probability is genuinely low — GitHub was burned publicly, reverted, and
  committed to notice. This is not an unmonitored risk; it is one with a
  six-month tripwire.
- The reactive fix is cheap and mechanical (below), on the order of minutes.
- It is what essentially every AUR package and Homebrew tap does. Self-hosting
  the tarball is the unusual choice, not the pinning.

The argument on the other side, recorded honestly: the failure is silent until
a user hits it, and it hits every version at once.

## If it recurs, the fix

Recompute the digests and republish both packages. The publishers already do
exactly this — they are the fix, not just the bump path:

- [`scripts/aur-publish.sh`](../scripts/aur-publish.sh) (kata 18cr)
- [`scripts/brew-publish.sh`](../scripts/brew-publish.sh) (kata nqmn)

Both take a `VERSION` and recompute from published artifacts, and both have a
`CHECK=1` mode that asserts a re-run for a published version is a no-op. The
by-hand recipes live in
[`packaging/aur/README.md`](../packaging/aur/README.md) and
[`packaging/brew/README.md`](../packaging/brew/README.md).

The cross-check to lean on: the AUR source package and the Homebrew formula pin
**the same tarball**, so they must carry **the same digest**. If those two ever
disagree, one of them is wrong.

## The alternative we did not take

Publish our own source tarball as a release asset and point both packages at
it, making the digest depend only on an artifact we publish:

1. `build.yml`: `git archive --format=tar.gz --prefix=tailport-$ver/ $GITHUB_REF_NAME`,
   uploaded as `tailport-$ver-src.tar.gz` with a `.sha256` beside it, exactly
   like the binaries already get.
2. Point `source=` and `url` at that asset.

This is the robust answer and remains available if the risk ever materializes
or the calculus changes. The trap to remember if it is ever adopted: `--prefix`
must keep matching what the PKGBUILD's `build()`/`check()`/`package()` `cd`
into (`$pkgname-$pkgver`), or every function breaks at once.
